**Outline**:
- intro: JavaScript has six primitive types â€” Symbol is the seventh. Unlike string or number, every Symbol() call produces a unique value. This lesson establishes the Symbol primitive before we use it to hook into built-in protocols.
- mechanic: `Symbol()` returns a unique primitive. ## 03-iterator-protocol

**Metadata preserved**: id: 03-iterator-protocol | type: koan | currency: stable | audio_value: high | estimated_minutes: 45 | framing_notes: pull vs push; IteratorResult design

**Outline**:
- intro: Before `for...of` existed, iterating JavaScript containers required index access or `.forEach()`. The iterator protocol formalizes a pull-based contract: the consumer calls `next()` and the producer yields one value at a time.
- mechanic: An iterator is any object with a `next()` method that returns `{ value: T, done: false }` or `{ value: TReturn, done: true }`. Optional `return(value?)` â€” called by `for...of` on early exit (break/throw); should do cleanup and return `{ value, done: true }`. Optional `throw(error)` â€” injects an error at the current suspension point.
- worked_example: Implement a `RangeIterator` manually (no generators) that iterates integers from `start` to `end`. Add a `return()` that logs cleanup. Show that `for...of` with a `break` calls `return()`.
- pitfalls: Forgetting `done: true` in the final `next()` call causes an infinite loop. Returning `undefined` from `next()` instead of a proper `IteratorResult`. Not implementing `return()` when the iterator holds external resources.
- exercise_setup: Koans test `IteratorResult` shapes, `done` semantics, and optional `return()`/`throw()` behavior.

**Sources**: stable.
- https://tc39.es/ecma262/#sec-iterator-interface
- https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Iteration_protocols

## 04-iterable-protocol

**Metadata preserved**: id: 04-iterable-protocol | type: koan | currency: stable | audio_value: medium | estimated_minutes: 40

**Outline**:
- intro: An iterator is stateful â€” it tracks position. An iterable is a factory for iterators: you can ask it for a fresh iterator as many times as you like. The distinction matters for reusability.
- mechanic: An iterable implements `[Symbol.iterator](): Iterator<T>`. `for...of` calls this method once, then drives the returned iterator. Spread and destructuring do the same. Arrays, strings, Maps, Sets, and generators are all built-in iterables. An iterator-iterable (returns `this` from `[Symbol.iterator]`) is iterable but only once.
- worked_example: Make the `RangeIterator` from lesson 03 into a proper `Range` iterable that can be iterated multiple times. Demonstrate that two separate `for...of` loops produce independent iterations.
- pitfalls: Making an iterator return `this` (making it also iterable) â€” it can only be iterated once. Forgetting that strings are iterable by code point, not byte.
- exercise_setup: Koans test `[Symbol.iterator]()` implementations, `Array.from()`, spread on custom iterables, multi-iteration semantics.

**Sources**: stable.
- https://tc39.es/ecma262/#sec-iterable-interface
- https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Iteration_protocols#the_iterable_protocol

## 05-generator-functions

**Metadata preserved**: id: 05-generator-functions | type: koan | currency: stable | audio_value: medium | estimated_minutes: 50

**Outline**:
- intro: Writing iterators by hand (lesson 03) is mechanical and error-prone. Generator functions are syntactic sugar that lets you write iterators as if they were synchronous coroutines â€” the runtime manages the state machine.
- mechanic: `function*` creates a generator function. Calling it returns a generator object (implements Iterator + Iterable). `yield expr` pauses execution and emits `expr` as `{ value: expr, done: false }`. Execution resumes on the next `next()` call. `yield*` delegates to another iterable. `next(inputValue)` makes `inputValue` the result of the `yield` expression inside the generator.
- worked_example: Rewrite `Range` from lesson 04 as a generator function. Then write `fibonacci()` as an infinite generator. Show `yield*` delegating to two sub-generators.
- pitfalls: Treating generator functions like regular functions (they don't execute on call). Mixing up `yield` and `return` (return ends the generator). Forgetting that `yield*` forwards the return value of the sub-generator as the expression value.
- exercise_setup: Koans test basic `yield`, infinite sequences, `yield*` delegation, and `next(value)` input.

**Sources**: stable.
- https://tc39.es/ecma262/#sec-generatorfunction-objects

## 06-generator-return-and-throw

**Metadata preserved**: id: 06-generator-return-and-throw | type: koan | currency: stable | audio_value: medium | estimated_minutes: 45

**Outline**:
- intro: Generators support bidirectional communication: callers can send values in via `next()`, force early termination via `return()`, and inject errors via `throw()`. Understanding these paths is critical for generators that hold resources.
- mechanic: `gen.return(value)` forces the generator to act as if it executed `return value` at the current yield point â€” triggers `finally` blocks. `gen.throw(error)` injects `error` at the current yield â€” it can be caught inside the generator. `try/finally` inside a generator runs the `finally` block even when `return()` or `throw()` is called externally.
- worked_example: A generator that opens a (mock) file handle and reads lines. Show that breaking out of a `for...of` loop triggers `return()`, which runs the `finally` block to close the handle.
- pitfalls: Assuming `return()` skips `finally` (it doesn't). Calling `throw()` on a done generator (throws the error at the call site). Using `try/catch` inside a generator to swallow errors from `throw()` when they should propagate.
- exercise_setup: Koans test `return()` + `finally`, `throw()` + catch inside generator, and cleanup ordering.

**Sources**: stable.

## 07-lazy-sequences

**Outline**:
- intro: Generators enable sequences that are never fully materialized in memory â€” useful for large datasets, infinite math sequences, and composing transformations without intermediate arrays.
- mechanic: An infinite generator never yields `done: true`. `take(n, iterable)` collects the first `n` items. Generator adapters â€” `map(fn, iterable)`, `filter(pred, iterable)` â€” transform a lazy sequence without materializing it. Composing adapters creates a pipeline evaluated only as the consumer pulls.

- exercise_setup: Koans test infinite sequences, lazy adapters, and pipeline composition.

## 08-async-iterator-protocol

**Metadata preserved**: id: 08-async-iterator-protocol | type: koan | currency: stable | audio_value: high | estimated_minutes: 45 | framing_notes: pull vs push async; compare to callbacks/events

**Outline**:
- intro: The sync iterator protocol assumes each `next()` is immediately available. Async iteration extends the protocol so that each `next()` can represent work that takes time â€” a database row, a network chunk, a file line.
- mechanic: `AsyncIterator` â€” `next()` returns `Promise<IteratorResult<T>>`. `Symbol.asyncIterator` â€” marks an object as async iterable. `for await...of` drives async iterators: calls `[Symbol.asyncIterator]()`, then `await`s each `next()`. `return()` and `throw()` are also async (return Promises). Node.js `Readable` streams and `ReadableStream` implement `Symbol.asyncIterator`.
- worked_example: Implement a `DelayedRange` that yields integers with a configured delay between each (simulating async data arrival). Iterate it with `for await...of` and show timing.
- pitfalls: Using `for...of` (sync) on an async iterable (silently iterates the iterator object itself, not the values). Forgetting to `await` the Promise returned by a manually called `next()`. Not cleaning up via `return()` on early exit.
- exercise_setup: Koans test async iteration, manual `next()` + await, and early exit cleanup.

## 09-async-generators

**Outline**:
- intro: Async generators combine the coroutine model of generators with the async/await model for Promises. They are the natural way to produce async iterables.
- mechanic: `async function*` creates an async generator function. `yield value` emits a value; the consumer awaits it. `await promise` inside the generator pauses for a Promise without emitting. **Outline**:
- intro: Managing cleanup â€” closing files, releasing locks, cancelling subscriptions â€” has always required manual `try/finally`. The explicit resource management proposal (TC39 stage 4, TypeScript 5.2+) adds `using` and `await using` as language-level cleanup declarations.
- mechanic: `Symbol.dispose` â€” implementing `[Symbol.dispose]()` makes an object synchronously disposable. `using x = resource` â€” calls `x[Symbol.dispose]()` at block exit (normal or exception). `Symbol.asyncDispose` + `await using` â€” the async variant. `DisposableStack` â€” composite disposal for multiple resources (LIFO order). - worked_example: Wrap a mock database connection with `Symbol.asyncDispose`. Use `await using conn = getConnection()` to ensure cleanup even if the query throws. Show with `DisposableStack` that multiple resources are released in reverse order.

- exercise_setup: Koans test `Symbol.dispose` implementation, `using` scope semantics, `await using`, `DisposableStack` ordering, and error suppression.
